## 1. 專案目的

VELOCUE 官網這一階段的目標，不是單純做一個漂亮首頁。

它的任務是先建立一套可長期沿用的網站設計標準，作為後續：

- 拉皮
- 內容整理
- 產品介紹頁擴充
- 功能逐步完整化
- 品牌一致性維護

的共用基礎。

這份文件是 VELOCUE 官網的總設計簡報與風格準則。

## 2. 產品定位

### 產品一句話

VELOCUE 是一個以路線為核心的騎前準備工具，幫助公路車、耐力與比賽取向騎士，把路線轉成可安排、可判斷、可輸出的騎乘筆記。

### 產品角色

它不是：

- 騎後分析平台
- 一般導航工具
- 路線記錄 App
- 休閒騎乘用 route viewer

它是：

- route briefing tool
- 基於路線做安排的筆記工具
- 結合地圖、等高線、註記與輸出的準備工具

### 核心承諾

VELOCUE 主要承諾的是：

`提升騎前準備品質`

次要價值是：

- 幫助更早做出配力判斷
- 幫助補給與會合安排更清楚
- 幫助重要提醒不只留在腦中

### 現階段定位

目前 cue strip / 列印輸出是主角。
但網站定位不能只等於列印工具。
VELOCUE 的上位概念必須是：

`把騎前準備做進路線裡`

## 3. 目標受眾

### 核心受眾

- 公路車騎士
- 耐力騎乘者
- 有活動 / 比賽情境的使用者
- 對補給、配力、爬坡與會合安排有需求的人

### 非核心受眾

- 路線可任意偏移的輕度休閒用戶
- 只想簡單看路線、不做安排的人
- 把產品期待成完整導航或訓練分析工具的人

### 受眾心態

這群人通常不是沒有路線，而是：

- 有路線，但還沒整理成騎乘安排
- 看得懂 GPX，但不夠直覺知道怎麼準備
- 知道比賽 / 活動要準備，但缺少一個能把判斷放進路線裡的工具

## 4. 品牌關鍵句

### 品牌主張

把騎前準備，直接做進路線裡。

### 品牌角色

VELOCUE 像一位冷靜、熟悉路線、懂比賽節奏的人，在出發前幫你把準備整理清楚。

### 品牌感受

- 溫暖
- 沉著
- 清楚
- 有準備感
- 可信
- 實用但不粗糙

## 5. 品牌語氣規範摘要

### 語氣方向

VELOCUE 的語氣應該是：

- 冷靜的
- 克制的
- 以準備為核心的
- 有路線與騎乘畫面的
- 原生繁中的

### 文案應該像什麼

- 在出發前，先把路線上的關鍵安排整理好。
- 不只看地圖，也把補給、會合與提醒放進路線裡。
- 讓騎前準備更完整，而不是到了路上才邊騎邊想。

### 文案不應該像什麼

- 釋放你的騎乘潛能
- 重新定義你的賽道掌控力
- 全面升級你的騎乘體驗
- 支援 GPX 上傳、註解、匯出功能

### 語氣規則

- 先講準備品質，再講表現
- 先講具體場景，再講抽象好處
- 可以有畫面，但不能過度文案化
- marketing 語言和 product UI 語言必須分開
- 中文是原生內容，不是英文翻譯層

## 6. 設計定位

### 設計一句話

VELOCUE 官網應該是：

`安靜、溫暖、寬鬆、成熟的工具感`

### 設計主張

它不是 dashboard 美學，也不是 lifestyle 形象頁。
它是一個實用工具，但呈現方式足夠有品牌質感。

### 核心設計張力

必須同時成立的幾件事：

- 工具是本體，品牌感是表達方式
- 留白很大，但資訊秩序更強
- 圓潤可以有，但不能可愛
- 暖色是情緒背景，不是功能色濫用
- 品牌敘事和產品證據要交錯出現

## 7. 視覺語言摘要

### 整體感受

- 日光感
- 空氣感
- 大留白
- 厚實圓潤的標題感
- 成熟穩定的工具結構

### 不應該出現的風格

- 後台模板感
- 運動科技廣告感
- 太冷的 SaaS 登陸頁
- 太可愛的圓體工具頁
- 太多發光、玻璃感、炫技漸層

### 色彩角色

- Base Canvas：暖白、米白、淺奶油白
- Surface：略有層次的區塊背景
- Ink：深灰、暖深灰、石墨灰
- Accent Primary：乾淨穩定的主 CTA 色
- Accent Warm：暖陽感點綴色
- Support Neutral：邊框、說明、次層結構色

### 字體角色

- Display：圓潤、厚實、現代，用於大標
- Text：穩定、耐讀、低個性，用於內文
- UI / Numeric：俐落、清楚，用於按鈕、標籤、數值

## 8. 首頁的核心任務

首頁不需要介紹所有功能。
首頁只要完成以下四件事：

1. 定義 VELOCUE 是什麼
2. 定義它是給誰用的
3. 說清楚它如何幫助騎前準備
4. 用產品證據建立可信度

## 9. 首頁敘事主軸

首頁的主敘事固定為：

`把騎前準備，直接做進路線裡。`

這比單講列印、等高線或比賽表現都更完整，也更能支撐未來擴充。

## 10. 建議首頁架構

### 1. Hero

任務：
先講主價值，建立品牌氣質與第一個產品印象。

內容應包含：

- 主標
- 副標
- 主 CTA
- 次 CTA
- 帶產品感的主視覺

### 2. Why This Exists

任務：
說清楚為什麼有路線還不等於準備完成。

### 3. What You Arrange On The Route

任務：
用補給、休息、會合、配力、爬坡提醒等情境，證明這是基於路線做安排的筆記工具。

### 4. Product Method

任務：
用簡單流程說明 VELOCUE 的工作方式。

建議流程：

1. 匯入路線
2. 對照地圖與等高線
3. 在關鍵位置加入註記
4. 輸出成可帶上路的形式

### 5. Product Evidence

任務：
展示產品畫面與輸出成果，建立可信度。

### 6. Why It Helps Before The Ride

任務：
把前面的功能與方法收斂回真實結果。

### 7. Output Today, Broader Briefing Tomorrow

任務：
說明 cue strip 是現在主角，但 VELOCUE 的定位更大。

### 8. Final CTA

任務：
邀請使用者開始整理一條路線。

## 11. Homepage Design Rules

### Hero 規則

- 大標應短、穩、有品牌感
- 不在 Hero 堆功能列表
- 視覺上要看得出路線 / 等高線 / 註記關係
- 不做過度 dashboard 拼貼

### Proof Block 規則

- 一段只證明一件事
- 圖像與文案互補，不重複
- 優先展示真實騎乘安排情境

### CTA 規則

主 CTA 建議語氣：

- 開始整理路線
- 開始做一條騎前筆記

次 CTA 建議語氣：

- 先看看怎麼安排
- 看看產品畫面

避免：

- 立即體驗
- 馬上開始
- 解鎖完整功能

## 12. 元件語言

### Button

- 穩
- 有重量
- 不促銷
- 不需要誇張 hover 效果

### Card

- 只用在內容分組需要時
- 不要整頁卡片海
- 卡片內部需短而清楚

### Badge / Label

- 俐落
- 簡潔
- 不要像可愛貼紙

### Screenshot Frame

- 可有柔和外框
- 不用厚重裝置 mockup
- 不可蓋過產品真實感

## 13. 產品畫面使用規則

- 一張圖只證明一件事
- 優先展示地圖、等高線、註記、輸出之間的關係
- 可以局部放大重點
- 不把最複雜的完整畫面直接丟上首頁
- 產品證據應早出現，但不能搶 Hero 敘事

## 14. 官網與產品 UI 的分工

### 官網負責

- 建立理解
- 建立氣質
- 建立信任
- 告訴使用者這個工具幫你提早想清楚什麼

### 產品 UI 負責

- 指引操作
- 承接功能
- 讓註記與輸出流程更直接

### 明確規則

- 不要把 UI 按鈕語言直接拿來當 hero 文案
- 不要讓官網長得像 dashboard
- 不要讓產品 UI 被品牌視覺過度情緒化

## 15. 設計與文案禁忌

### 文案禁忌

- 過度熱血
- 過度英文翻譯感
- 過度運動品牌口號
- 過度功能說明口吻

### 視覺禁忌

- 過多高彩度
- 過多漸層與玻璃感
- 過多小卡片、小圖示、小 badge
- 過可愛圓體
- 過度像 SaaS 模板

## 16. 審查清單

每次做 VELOCUE 官網內容或設計時，都檢查：

1. 這一版有沒有明確回到「騎前準備品質」？
2. 是否清楚表達這是給公路 / 耐力 / 比賽情境使用者？
3. 文案是否像原生繁中，而不是翻譯稿？
4. 是否同時有品牌感與工具感？
5. 圓潤感是否開始變可愛？
6. 暖色是否開始干擾結構？
7. 首頁是否先說價值，再說方法，再給證據？
8. 這套設計是否能延伸到未來頁面？

## 17. 這份 brief 的直接用途

這份文件可以直接用來作為：

- 官網首頁設計 brief
- 視覺參考收斂標準
- 文案方向校對標準
- 未來 style guide 的總摘要
- 之後實作網站時的設計審查依據

## 18. 下一步建議

到這裡，VELOCUE 官網設計系統的基礎已經完整。

後續最合理的路線有三種：

1. 開始做首頁 wireframe / section 草圖
2. 開始收斂實際字體與色票提案
3. 直接進入首頁視覺稿或前端實作

如果以建設順序來看，我建議下一步先做：

`首頁 wireframe + section-level content brief`

這樣後面不管你是自己設計、讓我幫你實作、還是找外部設計師，都會更穩。
